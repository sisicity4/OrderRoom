# OrderRoom 役割分担

更新日: 2026-07-18

期限とSliceは[ロードマップ](%E3%83%AD%E3%83%BC%E3%83%89%E3%83%9E%E3%83%83%E3%83%97.md)を正とする。この資料は、誰が何を確認し、どの状態で次の担当へ渡すかを管理する。下の期間表は元のロードマップを転記したもので、独自の期限を追加・変更しない。

## 1. メンバー

| メンバー | 役割 | 主な責任 |
| --- | --- | --- |
| taiga | PM | 仕様、スコープ、優先順位、進捗、完成判定 |
| なり | FE | 画面、React実装、API接続、スマホUI |
| えのもと | BE | API、DB、認証、集計、精算、API配信 |
| tomoya | BEサポート・QA | バックエンド実装支援、APIテスト、実機導線、受け入れ確認 |

## 2. 担当範囲

### taiga（PM）

- ロードマップの期限、Slice、縮退判断を管理する
- MVP、要件、画面、APIの用語が一致しているか確認する
- 新しい要望をMVPへ混ぜず、将来候補へ送る
- 1日1回、縦方向の導線がどこまで通るか確認する
- P0バグと期限への影響を整理する
- デモシナリオ、README、発表資料をまとめる

PMは実装ファイルの有無ではなく、受け入れ条件を完遂できたかで進捗を更新する。

### なり（FE）

- 作成、共有、参加、提案、ホスト、買い物、精算画面を実装する
- APIクライアントと環境変数を管理する
- participant tokenとhostKeyを仕様どおり保存・送信する
- 読込中、空、通信失敗、404、403、二重送信を扱う
- 伝票デザインを維持し、375px以上の主要幅を確認する
- 見積額、実購入額、負担額、立替額を区別して表示する

### えのもと（BE）

- Room、Participant、Item、Settlementのデータ整合性を管理する
- REST API、バリデーション、共通エラーを実装する
- participant tokenとhostKeyによる権限を実装する
- 見積、予算、均等負担、立替精算を決定的に計算する
- 外部DB認証情報なしで実行できる自動テストを用意する
- Docker、CORS、Neon接続を含む本番APIを整える

### tomoya（BEサポート・QA）

- えのもとの設計方針に合わせてバックエンド実装を支援する
- RepositoryモックまたはTestcontainersを使うテスト環境を支援する
- DTO、バリデーション、共通エラー、Controllerテストを切り出して担当する
- participant token、hostKey、他ルームIDの権限テストを担当する
- 集計と精算の端数・複数購入者・ロックのテストを担当する
- Sliceごとの受け入れ手順と実機確認を支援する
- P0、P1、P2で不具合を分類する

tomoyaは独自のAPI仕様を追加せず、[DB・API設計書](DB_API%E8%A8%AD%E8%A8%88%E6%9B%B8.md)と、えのもとの実装方針に合わせる。担当を分けるときは、同じControllerやServiceを同時に編集しないようファイルまたはエンドポイント単位で境界を決める。

## 3. 期間別の主担当

| 7/30〜8/4 | 精算・端数確認 | 購入額・精算 | 精算データ・計算API | 精算計算、端数、立替、ロック支援 |

## 4. Sliceの受け渡し

### PMからFE・BEへ

- 対象Sliceの入力、出力、権限、エラー、受け入れ条件を一つに決める
- 未決事項を実装担当へ判断させたままにしない
- 変更する場合は影響する資料とSliceを明記する

### BEからFEへ

- Method、Path、Header、Request、Response、Errorを共有する
- token、hostKeyを返す場所と返さない場所を明記する
- 未実装、仮実装、確定を区別する

### FEからQAへ

- 確認URL、必要データ、通常操作、異常操作を共有する
- 画面幅と対象ブラウザを示す
- 未対応状態を隠さず伝える

### QAからPMへ

- 再現手順、期待結果、実際の結果、重大度を記録する
- 「使いにくい」と「仕様と違う」を分ける
- P0は当日の完成判定へ反映する

## 5. 完了報告の共通形式

各担当は次の5点を揃える。

1. 完了した利用者操作
2. 確認した受け入れ条件
3. 未完了または仮実装
4. 次の担当が必要とする情報
5. 期限へ影響するリスク

## 6. PMの日次チェック

- [ ] 作成からどこまで通ったか
- [ ] FEとBEの契約が一致しているか
- [ ] 画面とAPIで同じ用語を使っているか
- [ ] P0バグが増えていないか
- [ ] MVP外の機能が混入していないか
- [ ] 翌日の担当と受け入れ条件が一つに決まっているか

# 役割分担

## 前提

- プロダクト名は「オーダールーム」。
- 主戦場は、イベント・飲み会・BBQ・宅飲みなどの買い出しや注文整理。
- MVPではリアルタイム同期を必須にしない。
- あまり機密性の高い情報は扱わず、イベント中に使いやすいことを優先する。
- 目安の期日は 2026-07-31。
- 7月後半まで余白を持たせ、途中で仕様調整や実装方針の見直しができるようにする。

## メンバー

| メンバー | 役割 | 主な担当 |
| --- | --- | --- |
| taiga | PM | 仕様整理、スコープ判断、優先順位決定、進捗管理 |
| なり | フロントエンド | 画面設計、React実装、スマホUIの調整 |
| えのもと | バックエンド | API設計、DB設計、Spring Boot実装 |
| tomoya | QA・調査 | 競合調査、テスト観点整理、ユーザー導線の確認 |

参考リンクは [参考リンク](%E5%8F%82%E8%80%83%E3%83%AA%E3%83%B3%E3%82%AF.md) にまとめる。

## 担当範囲

### taiga（PM）

- MVPの完成条件を管理する。
- 仕様が広がりすぎたときに、今回作るものと作らないものを判断する。
- 画面・API・DBの優先順位を決める。
- 発表や共有に必要な説明資料をまとめる。

### なり（フロントエンド）

- ルーム作成、参加、注文入力、注文一覧、集計表示の画面を作る。
- スマホで迷わず使えるUIを優先して調整する。
- ホスト用管理URLで開いたときの採用・却下・購入チェック画面を作る。
- APIとの接続部分を実装する。

### えのもと（バックエンド）

- ルーム、参加者、アイテムのDB設計を固める。
- REST APIを実装する。
- `host_key` を使ったホスト用管理URLの仕組みを実装する。
- 集計値を保存せず、提案データから計算して返す。

### tomoya（QA・調査）

- イベント利用を想定したテストケースを作る。
- 参加者URLとホスト用管理URLの導線を確認する。
- 競合サービスや近い使い方を調べ、差分を整理する。
- MVP完成条件に沿って、動作確認リストを作る。

## 最初に決めること

- ルーム作成時に入力する項目。
- 参加者が入力できる項目。
- ホストだけが操作できる項目。
- 集計画面に必ず出す項目。
- 発表または提出までに作る画面の優先順位。
