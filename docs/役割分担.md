## 2. 担当範囲

### taiga（PM）

- ロードマップの期限、Slice、縮退判断を管理する。
- MVP、要件、画面、APIの用語が一致しているか確認する。
- 新しい要望をMVPへ混ぜず、将来候補へ送る。
- 1日1回、縦方向の導線がどこまで通るか確認する。
- P0バグと期限への影響を整理する。
- 担当Issueの進捗を確認し、必要時に再配分を判断する。

### なり（FE）

- 作成、共有、参加、提案、ホスト、買い物、精算画面を実装する。
- API接続前は仮データで画面状態と操作導線を実装する。
- API契約確定後に仮データをAPIクライアントへ置き換える。
- participant tokenとhostKeyを仕様どおり保存・送信する。
- 読込中、空、通信失敗、404、403、二重送信を扱う。

### えのもと（BE）

- Room、Participant、Item、Settlementのデータ整合性を管理する。
- REST API、バリデーション、共通エラーを実装する。
- participant tokenとhostKeyによる権限を実装する。
- 外部DB認証情報なしで実行できる自動テストを用意する。
- BEからFEへ、Method、Path、Header、Request、Response、Errorを共有する。

### tomoya（今村、QA・BE支援）

- PMまたはBE主担当が切り出したIssue単位のQA・BE支援を担当する。
- 手動テスト、URL導線、権限、異常系の確認結果をIssueへ記録する。
- BE実装を支援する場合は、対象ファイルまたはエンドポイントの担当境界を事前に決める。
- 完了できない、または期限への影響が見込まれる場合は、現状と引継ぎ情報をIssueへ記録する。

## 3. 担当変更・引継ぎ

- MVPに関係する作業はGitHub Issueで管理し、担当者、期限、Slice、完了条件を記載する。
- 担当者が期限内の対応が難しいと判断した場合は、期限前にPMへ共有する。
- PMが進捗報告または着手を確認できず、期限へ影響すると判断した場合は、担当を再配分できる。
- 引継ぎ時は、Issueへ現在の状況、未完了項目、作業ブランチ、確認方法、詰まっている点を残す。
- 引継ぎは個人評価ではなく、MVP期限を守るための進行管理として扱う。

## 4. Sliceの受け渡し

### PMからFE・BEへ

- 対象Sliceの入力、出力、権限、エラー、受け入れ条件を一つに決める。
- 未決事項を実装担当へ判断させたままにしない。
- 変更する場合は影響する資料とSliceを明記する。

### BEからFEへ

- Method、Path、Header、Request、Response、Errorを共有する。
- token、hostKeyを返す場所と返さない場所を明記する。
- 未実装、モックUI、API実装済み、結合済み、確認済みを区別する。

### FEからQAへ

- 確認URL、必要データ、通常操作、異常操作を共有する。
- 画面幅と対象ブラウザを示す。
- 未対応状態を隠さず伝える。

### QAからPMへ

- 再現手順、期待結果、実際の結果、重大度を記録する。
- 「使いにくい」と「仕様と違う」を分ける。
- P0は当日の完成判定へ反映する。

## 5. 完了報告の共通形式

各担当は次の内容をIssueまたはPRへ残す。

1. 対応したGitHub IssueとPRのURL
2. 完了した利用者操作
3. 確認した受け入れ条件
4. 未完了、仮実装、モックUI
5. 次の担当が必要とする情報
6. 期限へ影響するリスク
