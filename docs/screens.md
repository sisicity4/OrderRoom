# OrderRoom 画面仕様

更新日: 2026-07-18

この資料は画面ルート、画面責務、表示状態、Figma反映事項、画面受け入れ条件を管理する。MVP範囲と実装順は[ロードマップ](%E3%83%AD%E3%83%BC%E3%83%89%E3%83%9E%E3%83%83%E3%83%97.md)を優先する。

## 1. デザイン方針

- ベージュの紙、黒い罫線、赤い印章を使った伝票表現を維持する
- 明朝系の大見出しと、等幅またはゴシック系の補助情報を組み合わせる
- モバイル中心の1カラムとする
- 赤い印章は主操作の視覚記号にする
- 色だけで提案中、採用、見送りを区別しない
- 375px幅で本文を12px未満にしない

## 2. 画面とルート

| 画面 | ルート | 利用者 | Slice |
| --- | --- | --- | --- |
| トップ | `/` | 全員 | 既存土台 |
| ルーム作成 | `/create` | ホスト | 1 |
| 作成成功 | `/rooms/{roomId}/created` | ホスト | 1 |
| 参加・通常ルーム | `/rooms/{roomId}` | 参加者 | 1〜2 |
| ホスト管理 | `/rooms/{roomId}/host?key=...` | ホスト | 3〜4 |
| 買い物 | `/rooms/{roomId}/shopping` | 全員 | 3・5 |
| 割り勘・精算 | `/rooms/{roomId}/settlement` | 全員 | 5 |

参加前と参加後は `/rooms/{roomId}` 内で、ルーム単位のparticipant tokenの有無により切り替える。汎用 `/join` で参加URLを再入力させない。

## 3. 共通状態

すべてのデータ取得画面で、次を用意する。

| 状態 | 表示と操作 |
| --- | --- |
| 読込中 | 操作対象を隠しすぎず、読み込み中と分かる表示を出す |
| 空 | 次に行う操作を説明する。例: 「まだ提案がありません」 |
| 通信失敗 | 原因を断定せず、再試行ボタンを出す |
| 404 | ルームが見つからないことと、トップへ戻る導線を出す |
| 403 | 操作権限がないことを示し、通常画面へ戻せるようにする |
| 入力不正 | 該当欄の近くに修正方法を出す |
| 送信中 | 二重送信を防ぎ、完了または失敗まで状態を維持する |

## 4. 画面別仕様

### 4.1 トップ

目的: OrderRoomの価値を短く説明し、ルーム作成へ案内する。

- 「ルームを作る」を主操作にする
- MVPではルーム一覧を置かない
- 汎用の伝票番号・参加URL入力を置かない

### 4.2 ルーム作成

入力:

- イベント名 `title`: 必須
- 開催日 `eventDate`: 任意
- メモ `memo`: 任意
- 予算上限 `budgetAmount`: 任意、0円以上

状態:

- 必須項目エラー
- 金額形式エラー
- 送信中
- API失敗と再試行

### 4.3 作成成功

- ルーム名を表示する
- 参加URLをコピーできる
- コピー完了を表示する
- ホストURLを参加URLと分けて表示する
- ホストURLへ「共有しない」警告を出す
- hostKeyを通常本文へ繰り返し表示しない
- 「参加者画面を開く」「幹事画面を開く」を区別する

### 4.4 参加・通常ルーム

参加前:

- ルーム名と開催日を表示する
- 表示名だけ入力する
- ログインやURL再入力を要求しない

参加後:

- 商品名、見積単価、数量、メモを提案できる
- 提案者名と状態を一覧表示する
- 自分の提案だけ編集・削除操作を表示する
- 手動更新と最終更新時刻を表示する
- rejectedは通常一覧で非表示にする

### 4.5 ホスト管理

- 幹事モードであることを常に表示する
- proposedへ採用・見送り操作を出す
- rejectedを確認できるフィルタを用意する
- 全提案を編集できる
- 管理URLの共有禁止警告を確認できる
- tokenやhostKeyを画面本文へ出さない

### 4.6 買い物

- acceptedだけを表示する
- 商品別数量を表示する
- ホストだけが購入済みを変更できる
- 統合MVPでは、購入済み商品へ「実購入額（この商品の合計）」と購入者を入力できる
- 見積単価と実購入額を別ラベルで表示する
- 実購入額または購入者が未入力なら精算前に警告する

### 4.7 集計

ホスト管理または買い物画面内の集計領域として表示する。

- 採用済み見積を最も目立つ位置へ出す
- 提案中見積を補助表示する
- 予算がある場合だけ残額または超過額を表示する
- 商品別数量はacceptedだけを対象にする
- rejectedを含めない
- 「合計金額」だけで意味を省略しない

### 4.8 割り勘・精算

- 精算対象者を選択する
- 実購入額合計を表示する
- 参加者別の負担額、立替額、差額を表示する
- 「AさんからBさんへ○円」の形式で送金案を表示する
- 参加者は自分が支払側の精算だけを精算済みにできる
- ホストは全件を精算済みにできる
- 精算済みがある場合、計算元がロックされていることを説明する
- ホストのリセットには確認ダイアログを出す
- 未入力商品、精算対象者0人、計算失敗を区別して表示する

## 6. 画面受け入れ条件

### 作成・参加

- 作成後に参加URLをコピーできる
- ホストURLは作成者だけに表示する
- 共有URLを直接開いてルーム名を確認できる
- 表示名だけで参加できる
- 2〜3タップ程度で参加後の画面へ到達できる

### 提案・ホスト

- 商品名、見積単価、数量、メモを入力できる
- 自分の提案だけ編集・削除できる
- 参加者が採用品を編集すると提案中へ戻る
- hostKeyなしでは採用、見送り、購入操作ができない

### 集計・精算

- 採用済み見積、提案中見積、実購入額、負担額を区別できる
- 予算未入力時は予算差分を表示しない
- 実購入額と購入者が揃わない商品を精算対象にしない
- 精算済み権限とロック状態を画面から理解できる

### 実機

- iPhone Safariで直接URL、コピー、フォーム、再読込を確認する
- 375px、390px、430pxで横スクロールなしに主操作を完了できる
- MacとiPhoneの2端末で手動更新による反映を確認する

## SP/PC

SP: スマートフォン用の画面
オーダー向けのwebアプリのため,スマートフォンから注文と確認がしやすいようなデザインにする.

PC: パソコン用の画面
パソコンからは、編集や細かい管理,確認などをしやすくデザインをする.
