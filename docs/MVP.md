# OrderRoom MVP仕様

更新日: 2026-07-18

この資料は、[ロードマップ](%E3%83%AD%E3%83%BC%E3%83%89%E3%83%9E%E3%83%83%E3%83%97.md)で定めたコア版と統合MVPの**機能範囲と完成シナリオ**を説明する。

期限、実装順、縮退判断はロードマップを優先する。画面の詳細は[画面仕様](screens.md)、APIとDBの詳細は[DB・API設計書](DB_API%E8%A8%AD%E8%A8%88%E6%9B%B8.md)を参照する。

## 概要

OrderRoomは、大学生のBBQ、宅飲み、旅行、学園祭などで、参加者が欲しい物を提案し、幹事が採用・見送りを決め、買い物、実購入額、均等割り、立替精算まで共有するログイン不要のWebアプリである。

相談や合意形成は外部チャット、提案から精算記録まではOrderRoom、実際の送金は外部の決済サービスで行う。

## MVPを二段階に分ける

### コア版: 2026-07-31

購入前の意思決定と買い物を完遂する。

> 作成 → 共有 → 参加 → 提案 → 編集 → 採用・見送り → 買い物 → 見積集計

### 統合MVP: 2026-08-07

コア版に購入後の精算記録を加える。

> 実購入額・購入者 → 精算対象者 → 均等割り → 立替精算案 → 精算済み記録

## 確定仕様

| 項目 | 仕様 |
| --- | --- |
| 対象 | 大学生のイベント幹事・参加者 |
| 参加方法 | 共有URLを開き、表示名だけ入力する |
| 参加URL | `/rooms/{roomId}`。`roomId`はUUID |
| ホストURL | `/rooms/{roomId}/host?key={hostKey}` |
| ルーム作成項目 | `title / eventDate / memo / budgetAmount` |
| `budgetAmount` | 任意、0円以上。未入力なら予算差分を表示しない |
| 商品項目 | `name / price / quantity / memo` |
| `price` | 見積単価。UIでも「見積単価」と表示する |
| API状態 | `proposed / accepted / rejected` |
| UI状態 | 提案中／採用／見送り |
| 見送りデータ | 削除せず保持し、通常画面では非表示にする |
| メイン合計 | acceptedの `price × quantity` の合計 |
| 補助合計 | proposedの `price × quantity` の合計 |
| 予算差分 | `budgetAmount - acceptedTotalPrice` |
| 購入チェック | acceptedだけ変更できる |
| 実購入額 | acceptedかつ購入済みの商品へ、その商品行の実際の支払総額を記録する |
| 購入者 | 購入済み商品ごとに一人を記録する |
| 精算対象者 | ホストが参加者から選ぶ。初期値は全参加者 |
| 割り勘方式 | 実購入額合計を精算対象者で均等に分ける |
| 端数 | 1円単位。参加登録順の早い人から1円ずつ加算する |
| 立替精算 | 各人の負担額と立替額から送金案を算出する |
| 精算記録 | 送金案ごとに未精算／精算済みを記録する |
| 同期 | リアルタイム同期は行わず、手動更新で再取得する |
| 決済 | 行わない。送金はアプリ外で行う |

## 権限

| 操作 | 参加者 | ホスト |
| --- | --- | --- |
| ルーム閲覧 | 可 | 可 |
| 商品提案 | 可 | 可 |
| 自分の提案編集・削除 | 可 | 可 |
| 他人の提案編集 | 不可 | 可 |
| 採用・見送り | 不可 | 可 |
| 購入済みチェック | 不可 | 可 |
| 実購入額・購入者の登録 | 不可 | 可 |
| 精算対象者の選択 | 不可 | 可 |
| 精算済み記録 | 自分が支払側の精算だけ可 | 全件可 |
| 精算のリセット | 不可 | 可 |

### 提案編集の再承認

- 参加者がacceptedの商品を編集した場合はproposedへ戻す
- ホストが編集した場合はacceptedを維持できる
- statusとpurchasedは通常の内容編集APIでは変更しない

## tokenとhostKey

- 参加成功時に参加者tokenを一度だけ返す
- FEはtokenをルーム単位で `localStorage` に保存する
- 参加者操作では `X-Participant-Token` を送る
- ホスト画面はURLのkeyを初回読込時に `sessionStorage` へ移す
- 可能なら初回読込後にkeyをアドレスバーから除去する
- ホスト操作では `X-Host-Key` を送る
- tokenとhostKeyは一覧レスポンス、通常画面、ログへ出さない
- 参加URLとホストURLを区別し、ホストURLには共有禁止の警告を出す

## 精算の固定ルール

- 見積額ではなく実購入額だけを使用する
- 実購入額または購入者が未入力の商品は計算対象外にし、警告を表示する
- 精算対象者が0人の場合は計算しない
- 各人の負担額合計は実購入額合計と一致させる
- 各人の立替差額合計は0円にする
- 同じ入力なら毎回同じ端数配分と送金案を返す
- 精算済みが1件でもある場合、実購入額、購入者、精算対象者を変更できない
- ホストが全精算をリセットした場合だけ、計算元を再編集できる

## MVPに含めないもの

- QRコード
- ルーム一覧、参加履歴一覧
- チャット、投票、通知
- AI機能、レシートOCR
- アプリ内決済、送金代行
- 重み付き割り勘、割合指定、個別商品の負担者指定
- 運転手割引、飲酒有無による自動調整
- 商品マスタ、店舗メニュー連携、モバイルオーダー連携
- リアルタイム同期

## 完成シナリオ

MacとiPhoneなど2台以上で、次を本番環境上で順番に完遂できること。

1. ホストがイベント名、開催日、メモ、任意の予算上限でルームを作る
2. 参加URLをコピーして共有する
3. 別の利用者が共有URLを直接開く
4. 表示名だけ入力して参加する
5. 商品名、見積単価、数量、メモを提案する
6. 自分の提案を編集・削除する
7. ホストが提案を採用または見送りにする
8. 採用品だけを買い物リストへ表示する
9. 採用品へ購入済みチェックを付ける
10. 採用済み見積、提案中見積、予算差分、商品別数量を表示する
11. 購入済み商品へ実購入額と購入者を記録する
12. 精算対象者を選び、均等割りの負担額を表示する
13. 複数購入者の立替額から送金案を表示する
14. 支払側本人またはホストが精算済みを記録する
15. 手動更新で別端末の変更を取得する
16. 空状態、読込中、通信失敗、存在しないルームを表示する

## 完成判定

- コア版は手順1〜10が完遂でき、P0バグが0件なら完成
- 統合MVPは手順1〜16が完遂でき、P0バグが0件なら完成
- ファイルや画面が存在するだけでは完成としない
- 詳細な画面受け入れ条件は[画面仕様](screens.md)を使う
- APIと計算の受け入れ条件は[DB・API設計書](DB_API%E8%A8%AD%E8%A8%88%E6%9B%B8.md)を使う

# MVP

## 概要

複数人の希望や注文を**ひとつのルームで集約し、誰が何をどれだけ欲しいか**を確認できるサービス。

---

## MVPで実現すること（スコープ）

### 1. ルーム作成（代表者）

- 注文ルームを新規作成できる
- 参加用のURL（必要ならQRコード）を発行・共有できる

### 2. ルーム参加（参加者）

- URL（またはQR）からルームに参加できる
- **名前のみ**入力して注文を開始できる（ログイン不要）

### 3. 注文登録（参加者）

- 以下を入力して注文を追加できる
    - 商品名
    - 価格
    - 数量
- 各注文に「追加した人（参加者名）」が紐づいて保持される

### 4. 自動集計

注文が追加・更新されたあと、画面表示時または更新操作時に以下を自動で計算・表示する（集計値は保存しない）。
リアルタイム同期はMVPでは必須にしない。

- 個人別の合計金額
- 全体の合計金額
- 商品ごとの合計数量

---

## MVPでは作らないもの（非スコープ）

- 決済
- リアルタイム同期
- モバイルオーダー連携
- アカウント登録 / ログイン
- クーポン
- 商品マスタ（メニュー管理）
- チャット

> 今回は **「注文をまとめる」** ことだけに集中する。
>

---

## データ設計（最小）

### 保存するデータ

- ルーム
- ユーザー（表示名のみでも可）
- 注文（商品名 / 価格 / 数量 / 追加者 / 作成日時 など）

### 計算で出すデータ（保存しない）

- 個人合計
- 全体合計
- 商品別数量

---

## 画面（最小）

- ルーム作成
- 参加（名前入力）
- 注文入力
- 注文一覧
- 集計（一覧と同画面でも可）

---

## 完成条件（MVPの成功基準）

- 代表者がルームを作成できる
- 参加者全員がルームに参加し、注文を入力できる
- 代表者が1画面で次を確認できる
    - 誰が何を注文したか
    - 全体で何を注文するか（商品別数量）
    - 合計金額
