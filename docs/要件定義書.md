# OrderRoom 要件定義書

更新日: 2026-07-18

> 「何を買う？」から「いくら払う？」までを、一つのルームで。

この資料はプロダクト要件と業務ルールを管理する。MVP範囲と期限は[ロードマップ](%E3%83%AD%E3%83%BC%E3%83%89%E3%83%9E%E3%83%83%E3%83%97.md)および[MVP仕様](MVP.md)を優先する。

## 1. プロダクト概要

| 項目 | 内容 |
| --- | --- |
| 名称 | OrderRoom（オーダールーム） |
| 形態 | ログイン不要のモバイル優先Webアプリ |
| 対象 | 大学生のBBQ、宅飲み、旅行、学園祭など |
| 利用者 | ホスト・幹事、参加者 |
| 目的 | 提案、採否、買い物、見積、実購入額、精算記録を一つのルームで管理する |

## 2. 解決する課題

イベント準備では、希望、予算、購入結果、精算情報がLINE、口頭、メモ、電卓へ分散しやすい。

- 誰が何を希望したか分からなくなる
- 買う物と見送る物が混ざる
- 幹事だけが会話を見返し、合計を計算する
- 重複購入や買い忘れが起きる
- 見積額と実際の支払額が混ざる
- 誰が立て替え、誰が誰へ払うか分かりにくい

OrderRoomは、相談そのものではなく、相談後の**提案 → 決定 → 買い物 → 精算記録**を管理する。

## 3. サービスの境界

| 領域 | 担当 |
| --- | --- |
| 相談、合意形成 | LINE、Discordなど外部チャット |
| 提案、採否、買い物、見積、精算記録 | OrderRoom |
| 実際の決済、送金 | 銀行、決済サービスなど外部手段 |

OrderRoomは店舗へ注文を送らず、送金も代行しない。

## 4. 利用者

### ホスト・幹事

- ルームを作成して参加URLを共有する
- 提案を採用または見送りにする
- 買い物の完了、実購入額、購入者を記録する
- 精算対象者を選択し、精算案を確認する
- 必要に応じて全精算をリセットする

### 参加者

- 共有URLから表示名だけで参加する
- 欲しい物を提案する
- 自分の提案を編集または削除する
- 採否、買い物状況、見積、精算案を確認する
- 自分が支払う精算を精算済みにする

## 5. 基本利用フロー

1. ホストがルームを作成する
2. 参加URLを共有する
3. 参加者が表示名だけで参加する
4. 参加者が欲しい物を提案する
5. ホストが採用または見送りを決める
6. 採用品を買い物リストとして使用する
7. 購入済み商品へ実購入額と購入者を記録する
8. 精算対象者を確定する
9. 均等負担と立替精算案を確認する
10. 実際の送金後、精算済みを記録する

## 7. 業務ルール

### 用語と状態

| API値 | UI表記 | 意味 |
| --- | --- | --- |
| `proposed` | 提案中 | ホストの判断前、または参加者編集後の再確認待ち |
| `accepted` | 採用 | 買い物対象 |
| `rejected` | 見送り | 保存はするが通常画面では非表示 |

UIでは「注文」「却下」「pending」を使用せず、「提案」「見送り」「proposed」に統一する。

### 提案と買い物

- 参加者は自分の提案だけ編集・削除できる
- ホストはすべての提案を編集できる
- 参加者が採用品を編集した場合はproposedへ戻す
- ホストが採用品を編集した場合はacceptedを維持できる
- accepted以外は購入済みにできない
- rejectedは履歴として保持し、集計から除外する

### 金額

| 金額 | 定義 |
| --- | --- |
| 見積単価 | 提案時の1個あたりの予定価格 |
| 採用済み見積 | acceptedの `見積単価 × 数量` の合計 |
| 提案中見積 | proposedの `見積単価 × 数量` の合計 |
| 実購入額 | 購入後、その商品行へ記録する実際の支払総額 |
| 負担額 | 精算対象者が均等に負担する金額 |
| 立替額 | 参加者が実際に支払った金額 |
| 精算額 | 立替差額を解消するために支払う金額 |

これらを画面やAPIで単に「合計金額」と表記しない。

### 精算

- 実購入額だけを計算に使う
- 端数は参加登録順の早い人から1円ずつ加算する
- 精算対象者の初期値は全参加者とする
- 精算済みが1件でもある間は計算元を変更できない
- ホストが全精算をリセットした後だけ再計算できる
- 実際の送金処理は行わない

## 既存サービスの課題

飲み会やパーティーでは、注文や買い出しの希望がLINEや口頭に散らばりやすい。

- 誰が何を欲しいと言ったか分からなくなる
- ホストだけがLINEを見返すことになる
- 買うものと見送るものが混ざる
- 合計金額を手計算する必要がある
- 買い忘れや重複購入が起きる
- レシートや精算が後から面倒になる

居酒屋のモバイルオーダーは「お店に注文する」ための仕組みで、割り勘アプリは「支払いを清算する」ための仕組みになりがち。

本アプリ（オーダールーム）は、その**前後をつなぐ“希望の整理・決定・買い物”**に焦点を当てる。

---

## 既存サービスとの比較（利点と差分）

### 1) 居酒屋のモバイルオーダー（店内注文システム）との違い

- **目的**: 店に料理を注文する（店のメニューが前提）
- **前提**: 来店済み・席/テーブルに紐づく・店側システムが必要
- **強み**: 注文が店舗に直接届く、提供管理と連携できる
- **弱み（本課題に対して）**:
    - 店外の「買い出し」や「何を買うかの相談」には使いづらい
    - 採用/却下など“意思決定”の管理が主目的ではない

→ **本アプリの差分**: 店やメニューに依存せず、イベント準備の「欲しいもの」整理から扱える。

### 2) 割り勘アプリ（精算アプリ）との違い

- **目的**: 支払い後の精算（誰がいくら払うか）を計算する
- **強み**: 支払いの計算が速い、立替/送金の整理に強い
- **弱み（本課題に対して）**:
    - そもそも「何を買うか」「誰が欲しいか」の情報が残りにくい
    - 買い物リスト化、購入チェック、採用/却下の流れは薄いことが多い

→ **本アプリの差分**: 精算より前の「提案 → 採用 → 買い物 → 集計」を一続きで管理できる。

### 3) 一般的なToDo/買い物リストアプリとの違い

- **目的**: タスクや買う物のチェック
- **強み**: シンプルで早い、チェック運用が得意
- **弱み（本課題に対して）**:
    - 誰が提案したか、個人別の金額、商品別の数量集計が弱い
    - 参加者が増えると「意思決定」と「集計」が分離しやすい

→ **本アプリの差分**: 参加者情報を前提に、集計（個人別/全体/商品別）まで一気通貫。

### 本アプリ（オーダールーム）の利点まとめ

- 参加者が**ログイン不要**で参加しやすい（イベント向け）
- 「提案」と「採用/却下」を分けて、**買うものを確定**できる
- 買い物リストと合計金額が同じルームで繋がり、**見返しが楽**
- 個人別/全体/商品別の集計があるため、**重複購入・買い忘れを減らせる**

---

## 改善コンセプト

**「何買う？」から「いくら払う？」までを、ひとつのルームで。**

オーダールームでは、参加者が欲しいものを自由に提案し、ホストが採用・却下を決める。

採用されたものだけが買い物リストになり、合計金額も自動で集計される。

---

## 設計の核

このアプリの中心は「注文」ではなく「ルーム」。

ルームの中で、以下の流れを管理する。

1. 参加者が欲しいものを提案する
2. ホストが採用・却下を決める
3. 採用されたものが買い物リストになる
4. 買ったものをチェックする
5. 合計金額を確認する

これにより、LINE・メモ・電卓・買い物リストを行き来しなくても、イベント準備を1画面で進められる。

## 非機能要件・制約

- スマホで使いやすいUIを優先する
- イベント中にすぐ使えるよう、ログインは不要にする
- 小規模利用を想定する
- リアルタイム性はMVPでは必須にしない
- 集計値は保存せず、提案データから計算する
- あまり機密性の高い情報は扱わない前提にする
- 厳密な認証より、イベント中に迷わず使える導線を優先する

## このアプリの強み

オーダールームは、単なる注文アプリではない。

飲食店のモバイルオーダーが「店に注文する」ためのものなら、このアプリは「みんなで何を買うか決める」ためのもの。

LINE、メモ、電卓、買い物リストに散らばるイベント準備を、1つのルームにまとめるところが最大の特徴である。
