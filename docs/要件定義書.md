# 要件定義書

> パーティー・飲み会・BBQなどのイベントで、「何を買う？」「誰が欲しい？」「いくらかかる？」を1つのルームで整理するWebアプリ。
>
>
> LINEで散らばる希望を、提案・採用・買い物リスト・集計までつなげる。
>

---

## 概要

| 項目 | 内容 |
| --- | --- |
| 名称 | オーダールーム |
| 形態 | Webアプリ |
| GitHub | [sisicity4/OrderRoom](https://github.com/sisicity4/OrderRoom) |
| 対象 | パーティー・飲み会・BBQ・宅飲み・買い出し |
| 主な利用者 | ホスト・幹事・参加者 |
| 目的 | イベント準備の注文・買い物・金額管理を楽にする |

---

## 既存サービスの課題

飲み会やパーティーでは、注文や買い出しの希望がLINEや口頭に散らばりやすい。

- 誰が何を欲しいと言ったか分からなくなる
- ホストだけがLINEを見返すことになる
- 買うものと見送るものが混ざる
- 合計金額を手計算する必要がある
- 買い忘れや重複購入が起きる
- レシートや精算が後から面倒になる

居酒屋のモバイルオーダーは「お店に注文する」ための仕組みで、割り勘アプリは「支払いを清算する」ための仕組みになりがち。

本アプリ（オーダールーム）は、その**前後をつなぐ“希望の整理・決定・買い物”**に焦点を当てる。

---

## 既存サービスとの比較（利点と差分）

### 1) 居酒屋のモバイルオーダー（店内注文システム）との違い

- **目的**: 店に料理を注文する（店のメニューが前提）
- **前提**: 来店済み・席/テーブルに紐づく・店側システムが必要
- **強み**: 注文が店舗に直接届く、提供管理と連携できる
- **弱み（本課題に対して）**:
    - 店外の「買い出し」や「何を買うかの相談」には使いづらい
    - 採用/却下など“意思決定”の管理が主目的ではない

→ **本アプリの差分**: 店やメニューに依存せず、イベント準備の「欲しいもの」整理から扱える。

### 2) 割り勘アプリ（精算アプリ）との違い

- **目的**: 支払い後の精算（誰がいくら払うか）を計算する
- **強み**: 支払いの計算が速い、立替/送金の整理に強い
- **弱み（本課題に対して）**:
    - そもそも「何を買うか」「誰が欲しいか」の情報が残りにくい
    - 買い物リスト化、購入チェック、採用/却下の流れは薄いことが多い

→ **本アプリの差分**: 精算より前の「提案 → 採用 → 買い物 → 集計」を一続きで管理できる。

### 3) 一般的なToDo/買い物リストアプリとの違い

- **目的**: タスクや買う物のチェック
- **強み**: シンプルで早い、チェック運用が得意
- **弱み（本課題に対して）**:
    - 誰が提案したか、個人別の金額、商品別の数量集計が弱い
    - 参加者が増えると「意思決定」と「集計」が分離しやすい

→ **本アプリの差分**: 参加者情報を前提に、集計（個人別/全体/商品別）まで一気通貫。

### 本アプリ（オーダールーム）の利点まとめ

- 参加者が**ログイン不要**で参加しやすい（イベント向け）
- 「提案」と「採用/却下」を分けて、**買うものを確定**できる
- 買い物リストと合計金額が同じルームで繋がり、**見返しが楽**
- 個人別/全体/商品別の集計があるため、**重複購入・買い忘れを減らせる**

---

## 改善コンセプト

**「何買う？」から「いくら払う？」までを、ひとつのルームで。**

オーダールームでは、参加者が欲しいものを自由に提案し、ホストが採用・却下を決める。

採用されたものだけが買い物リストになり、合計金額も自動で集計される。

---

## 設計の核

このアプリの中心は「注文」ではなく「ルーム」。

ルームの中で、以下の流れを管理する。

1. 参加者が欲しいものを提案する
2. ホストが採用・却下を決める
3. 採用されたものが買い物リストになる
4. 買ったものをチェックする
5. 合計金額を確認する

これにより、LINE・メモ・電卓・買い物リストを行き来しなくても、イベント準備を1画面で進められる。

---

## 機能要件

| # | 機能 | 概要 |
| --- | --- | --- |
| F1 | ルーム作成 | ホストがイベント用のルームを作成する |
| F2 | ルーム参加 | URLまたはQRコードから参加する |
| F3 | 名前入力 | ログインなしで名前を入力して参加する |
| F4 | 欲しいもの提案 | 参加者が商品名・数量・価格・メモを入力する |
| F5 | 採用・却下 | ホストが提案を採用または却下する |
| F6 | 買い物リスト | 採用されたものだけを一覧化する |
| F7 | 購入チェック | 買ったものにチェックを付ける |
| F8 | 自動集計 | 合計金額・商品数・数量を自動計算する |
| F9 | 参加者別表示 | 誰が何を提案したか確認できる |
| F10 | ルーム共有 | 招待URLをコピーできる |
| F11 | ホスト用管理URL | 採用・却下・購入チェックを行うための管理URLを発行する |

---

## 非機能要件・制約

- スマホで使いやすいUIを優先する
- イベント中にすぐ使えるよう、ログインは不要にする
- 小規模利用を想定する
- リアルタイム性はMVPでは必須にしない
- 集計値は保存せず、提案データから計算する
- あまり機密性の高い情報は扱わない前提にする
- 厳密な認証より、イベント中に迷わず使える導線を優先する

---

## MVPで作らないもの

最初の開発では、以下は実装しない。

- 決済機能
- レシートOCR
- 割り勘の完全自動化
- チャット機能
- 通知機能
- 商品マスタ
- 店舗メニュー連携
- モバイルオーダー連携
- AIおすすめ機能

MVPでは、**提案 → 採用 → 買い物リスト → 合計金額表示**に集中する。

## テーブル構造

| テーブル | 主なカラム |
| --- | --- |
| rooms | id / title / date / memo / host_key / created_at |
| participants | id / room_id / name / created_at |
| items | id / room_id / participant_id / name / price / quantity / memo / status / purchased / created_at |

### status

| 値 | 意味 |
| --- | --- |
| proposed | 提案中 |
| accepted | 採用 |
| rejected | 却下 |

### ホスト権限

- MVPではログインを作らず、ルームURLとは別にホスト用管理URLを発行する。
- `host_key` はホスト用管理URLに含める管理キーとして扱う。
- ホスト用管理URLを知っている人は、採用・却下・購入チェックを操作できる。
- 機密性よりイベント中の使いやすさを優先し、厳密な本人確認はMVP後の検討に回す。

## 開発日数の目安（MVP）

前提: 1人開発、最低限のデザイン、テストは簡易（手動中心）

### フェーズ別

- 画面と導線の確定（要件の固め）: 0.5〜1日
- DB設計 + API設計: 1日
- バックエンド実装（ルーム/参加者/アイテムCRUD、集計）: 3〜5日

- 仕上げ（入力バリデーション、UI微調整、デプロイ）: 1〜2日

**合計: 8〜14日（約1.5〜3週間）**

- 仕様変更・手戻り: +2〜4日

## このアプリの強み

オーダールームは、単なる注文アプリではない。

飲食店のモバイルオーダーが「店に注文する」ためのものなら、このアプリは「みんなで何を買うか決める」ためのもの。

LINE、メモ、電卓、買い物リストに散らばるイベント準備を、1つのルームにまとめるところが最大の特徴である。
