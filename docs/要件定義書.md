# OrderRoom 要件定義書

更新日: 2026-07-18

> 「何を買う？」から「いくら払う？」までを、一つのルームで。

この資料はプロダクト要件と業務ルールを管理する。MVP範囲と期限は[ロードマップ](%E3%83%AD%E3%83%BC%E3%83%89%E3%83%9E%E3%83%83%E3%83%97.md)および[MVP仕様](MVP.md)を優先する。

## 1. プロダクト概要

| 項目 | 内容 |
| --- | --- |
| 名称 | OrderRoom（オーダールーム） |
| 形態 | ログイン不要のモバイル優先Webアプリ |
| 対象 | 大学生のBBQ、宅飲み、旅行、学園祭など |
| 利用者 | ホスト・幹事、参加者 |
| 目的 | 提案、採否、買い物、見積、実購入額、精算記録を一つのルームで管理する |

## 2. 解決する課題

イベント準備では、希望、予算、購入結果、精算情報がLINE、口頭、メモ、電卓へ分散しやすい。

- 誰が何を希望したか分からなくなる
- 買う物と見送る物が混ざる
- 幹事だけが会話を見返し、合計を計算する
- 重複購入や買い忘れが起きる
- 見積額と実際の支払額が混ざる
- 誰が立て替え、誰が誰へ払うか分かりにくい

OrderRoomは、相談そのものではなく、相談後の**提案 → 決定 → 買い物 → 精算記録**を管理する。

## 3. サービスの境界

| 領域 | 担当 |
| --- | --- |
| 相談、合意形成 | LINE、Discordなど外部チャット |
| 提案、採否、買い物、見積、精算記録 | OrderRoom |
| 実際の決済、送金 | 銀行、決済サービスなど外部手段 |

OrderRoomは店舗へ注文を送らず、送金も代行しない。

## 4. 利用者

### ホスト・幹事

- ルームを作成して参加URLを共有する
- 提案を採用または見送りにする
- 買い物の完了、実購入額、購入者を記録する
- 精算対象者を選択し、精算案を確認する
- 必要に応じて全精算をリセットする

### 参加者

- 共有URLから表示名だけで参加する
- 欲しい物を提案する
- 自分の提案を編集または削除する
- 採否、買い物状況、見積、精算案を確認する
- 自分が支払う精算を精算済みにする

## 5. 基本利用フロー

1. ホストがルームを作成する
2. 参加URLを共有する
3. 参加者が表示名だけで参加する
4. 参加者が欲しい物を提案する
5. ホストが採用または見送りを決める
6. 採用品を買い物リストとして使用する
7. 購入済み商品へ実購入額と購入者を記録する
8. 精算対象者を確定する
9. 均等負担と立替精算案を確認する
10. 実際の送金後、精算済みを記録する

## 7. 業務ルール

### 用語と状態

| API値 | UI表記 | 意味 |
| --- | --- | --- |
| `proposed` | 提案中 | ホストの判断前、または参加者編集後の再確認待ち |
| `accepted` | 採用 | 買い物対象 |
| `rejected` | 見送り | 保存はするが通常画面では非表示 |

UIでは「注文」「却下」「pending」を使用せず、「提案」「見送り」「proposed」に統一する。

### 提案と買い物

- 参加者は自分の提案だけ編集・削除できる
- ホストはすべての提案を編集できる
- 参加者が採用品を編集した場合はproposedへ戻す
- ホストが採用品を編集した場合はacceptedを維持できる
- accepted以外は購入済みにできない
- rejectedは履歴として保持し、集計から除外する

### 金額

| 金額 | 定義 |
| --- | --- |
| 見積単価 | 提案時の1個あたりの予定価格 |
| 採用済み見積 | acceptedの `見積単価 × 数量` の合計 |
| 提案中見積 | proposedの `見積単価 × 数量` の合計 |
| 実購入額 | 購入後、その商品行へ記録する実際の支払総額 |
| 負担額 | 精算対象者が均等に負担する金額 |
| 立替額 | 参加者が実際に支払った金額 |
| 精算額 | 立替差額を解消するために支払う金額 |

これらを画面やAPIで単に「合計金額」と表記しない。

### 精算

- 実購入額だけを計算に使う
- 端数は参加登録順の早い人から1円ずつ加算する
- 精算対象者の初期値は全参加者とする
- 精算済みが1件でもある間は計算元を変更できない
- ホストが全精算をリセットした後だけ再計算できる
- 実際の送金処理は行わない

## 8. 非機能要件

| ID | 分類 | 要件 |
| --- | --- | --- |
| N01 | 対応端末 | iPhoneを優先し、Macの主要ブラウザでも利用できる |
| N02 | 画面幅 | 375px、390px、430pxで主要操作が崩れない |
| N03 | 参加性 | アカウント登録なし、2〜3タップ程度でルームへ到達する |
| N04 | 同期 | リアルタイムを前提とせず、手動更新を明示する |
| N05 | 秘密情報 | DB認証情報、participant token、hostKeyを画面本文やログへ出さない |
| N06 | 権限 | 参加者操作とホスト操作を秘密値で分離する |
| N07 | エラー | 通信失敗、404、入力不正、権限不足から復帰できる |
| N08 | 整合性 | 集計値は保存せず計算し、精算合計の一致を自動テストする |
| N09 | アクセシビリティ | 色だけで状態を区別せず、本文を12px未満にしない |
| N10 | デプロイ | FE、API、DBをHTTPSで接続し、許可Originを限定する |

## 9. 画面要件

| 画面 | 主な目的 |
| --- | --- |
| トップ | サービス説明とルーム作成導線 |
| ルーム作成 | イベント情報と予算上限の登録 |
| 作成成功 | 参加URLの共有とホストURLの注意表示 |
| 参加・通常ルーム | 表示名参加、提案、一覧、手動更新 |
| ホスト管理 | 採用、見送り、買い物、集計 |
| 買い物 | acceptedの購入確認、実購入額・購入者入力 |
| 精算 | 対象者、負担額、立替額、送金案、精算状態 |

画面ルートと状態は[画面仕様](screens.md)を参照する。

## 10. データ領域

保存するデータ:

- ルーム
- 参加者と本人確認token
- 提案商品と状態
- 精算対象者
- 送金案と精算済み日時

計算するデータ:

- 採用済み見積、提案中見積、予算差分
- 商品別数量、参加者別の提案件数・採用見積
- 実購入額合計、負担額、立替額、送金案

テーブルとAPIは[DB・API設計書](DB_API%E8%A8%AD%E8%A8%88%E6%9B%B8.md)を参照する。

## 12. 非スコープと将来候補

MVPの非スコープは[MVP仕様](MVP.md#MVPに含めないもの)を正とする。

完成後に検討する候補:

- レシートOCR
- 過去イベントの再利用
- イベントテンプレート
- 人数に応じた買い物量の提案
- 重み付き割り勘
- 通知、リアルタイム同期
- 厳密なアカウント認証
