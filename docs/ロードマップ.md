## 1. この資料の役割

このロードマップは、OrderRoomの**優先順位、期限、実装順、完了判定**を管理する主資料である。

機能や画面、APIの詳細は各専門資料へ分割する。専門資料がこのロードマップのMVP範囲や期限と矛盾する場合は、ロードマップを優先し、専門資料側を更新する。

## 2. 結論と期限

OrderRoomは、購入前の「提案・採否・買い物」と、購入後の「実購入額・均等割り・立替精算」を一つのルームで扱う。

- 2026-07-31: 購入前導線を完成させたコア版
- 2026-08-07: 精算まで完遂できる統合MVP
- 2026-08-14: UI、利用検証、説明資料まで揃えたv1.0

AI、OCR、アプリ内決済はこの期間へ追加しない。

2026年7月16日は作業日に数えず、実装・仕様反映は7月17日から開始する。資料再整理によって元の期限やSlice期間は変更しない。

## 5. 実装ロードマップ

### Slice 0: 仕様固定

期限: 7月17日

主担当: PM。FE、BE、QAが各担当領域を確認する。

- [ ] ロードマップと各専門資料の用語を統一する
- [ ] 対象を大学生のイベント利用へ統一する
- [ ] QR、一覧、AI、OCR、アプリ内決済をMVP外へ置く
- [ ] 予算上限と予算差分をコア版へ含める
- [ ] 均等割り、立替額、送金案、精算済み記録を統合MVPへ含める
- [ ] API状態を `proposed / accepted / rejected` に固定する
- [ ] UI表記を「提案中／採用／見送り」に固定する
- [ ] Figmaの不足画面と不要導線を画面仕様へ反映する
- [ ] 各担当が自分の次の受け入れ条件を説明できる

完了条件: 各専門資料から旧仕様のQR必須、Vercel採用、`pending`、participantIdだけの本人確認、精算MVP外という記述がなくなる。

### Slice 1: 作成・共有・参加

期限: 7月18日〜7月21日

依存: Slice 0

BE:

- `GET /api/rooms/{roomId}`
- `POST /api/rooms/{roomId}/participants`
- Roomへ任意の `budgetAmount` を追加
- Participantへtokenを追加し、参加時に発行
- CORSと開発用proxyを設定
- 外部DB認証情報なしで実行できるテスト環境を用意

FE:

- APIクライアントと `VITE_API_BASE_URL` を追加
- CreatePageをルーム作成APIへ接続
- 作成成功、参加URLコピー、ホストURL警告を実装
- `/rooms/:roomId`を実装
- ルーム名、開催日、予算を表示
- 表示名入力と参加APIを接続
- 参加者tokenをルーム単位で保存

受け入れ条件:

- Macでルームを作成できる
- iPhoneで参加URLを直接開ける
- 表示名だけで同じルームへ参加できる

### Slice 2: 商品提案

期限: 7月22日〜7月24日

依存: Slice 1

BE:

- 商品の作成・一覧・編集・削除API
- 参加者操作で `X-Participant-Token` を検証
- 自分の提案だけ編集・削除可能にする
- 初期値を `price=0 / quantity=1 / status=proposed / purchased=false` とする

FE:

- 商品名、見積単価、数量、メモの提案フォーム
- 提案一覧、提案者名、提案中ラベル
- 自分の提案の編集・削除
- 読込中、空状態、通信失敗、手動更新、最終更新時刻

受け入れ条件: 別の参加者の提案は編集・削除できず、自分の提案だけ操作できる。

### Slice 3: ホスト判断・買い物

期限: 7月25日〜7月27日

依存: Slice 2

BE:

- `X-Host-Key`によるホスト確認
- 採用・見送り、購入済み変更API
- accepted以外の購入チェックを拒否
- rejectedを削除せず保持
- 参加者が採用品を編集した場合はproposedへ戻す

FE:

- 幹事モード、採用・見送り操作
- 見送り一覧フィルタ
- acceptedだけの買い物リスト
- 購入済みチェック
- 管理URLの共有禁止警告

受け入れ条件: hostKeyを持たない利用者はホスト操作ができない。

### Slice 4: 集計・予算

期限: 7月28日〜7月29日

依存: Slice 3

BE:

- 採用済み見積、提案中見積、予算差分、商品別数量を返す
- rejectedを集計から除外
- 集計値を保存せず都度算出

FE:

- 採用済み見積を主表示
- 提案中見積を補助表示
- 残り予算または予算超過額を表示
- 商品別数量と参加者別の提案件数・採用見積を表示

受け入れ条件: 見積額が実購入額や精算額として表示されない。

### Slice 5: 実購入額・割り勘・立替精算

期限: 7月30日〜8月4日

依存: Slice 4

BE:

- 購入済み商品へ実購入額と購入者を保存
- 精算対象者を保存
- 均等負担、立替差額、送金案を算出
- 1円の端数を参加登録順で配分
- 精算済み記録とリセットを実装
- 計算元、負担額、立替額、送金額の整合テストを作成

FE:

- 実購入額と購入者の入力
- 精算対象者の選択
- 参加者別の負担額、立替額、差額
- 「AさんからBさんへ○円」の送金案
- 精算済み操作とホストのリセット確認
- 未入力商品、対象者0人、変更ロックのエラー表示

受け入れ条件: 見積ではなく実購入額だけで計算され、負担額合計と実購入額合計が一致する。

### Slice 6: デプロイ・結合・QA

期限: 8月5日〜8月7日

依存: Slice 1〜5

- Cloudflare PagesへFEを配信
- RenderへDocker方式でAPIを配信
- Neon本番接続とSSLを確認
- `/rooms/{uuid}` の直接アクセスを確認
- 本番環境変数とCORSを確認
- 不正なroomId、token、hostKeyを確認
- 通信失敗、二重送信、再読込を確認
- 割り勘の端数、複数購入者、精算権限を確認
- MacとiPhoneの2端末で手動更新を確認
- 375px、390px、430px幅を確認
- READMEへ起動方法、構成、デモURLを記載

受け入れ条件: 本番環境で統合MVPの完成シナリオを最初から最後まで完遂できる。
