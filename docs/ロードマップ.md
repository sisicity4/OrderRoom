## 2. プロダクト定義

OrderRoomは、大学生のBBQ、宅飲み、旅行、学園祭などで、参加者が欲しい物を提案し、幹事が採用・見送りを決め、買い物リスト、実購入額、割り勘、立替精算まで共有するログイン不要のWebアプリである。

### 価値の中心

- アプリのインストールとアカウント登録が不要
- 共有URLから名前だけで参加できる
- 誰が何を希望したかが分かる
- 幹事の判断結果が一つの注文票に残る
- 採用品だけを買い物リストにできる
- 採用済み見積と予算との差を購入前に確認できる
- 誰がいくら立て替え、誰が誰へいくら払うかを確認できる
- 精算済みかどうかをルーム内へ記録できる

### OrderRoomが扱わないこと

- LINE・Discordで行う相談や合意形成
- クレジットカードや送金サービスによる実際の決済
- 複雑な傾斜配分、割合指定、運転手割引

相談は外部チャット、提案から精算記録まではOrderRoom、実際の送金は銀行・決済サービス、という境界にする。

## 5. MVP確定仕様

| 項目 | 確定仕様 |
| --- | --- |
| 対象 | 大学生のイベント幹事・参加者 |
| 参加方法 | 共有URLを開き、表示名だけ入力する |
| QRコード | MVP外 |
| ルーム一覧 | MVP外 |
| 参加URL | `/rooms/{roomId}`。`roomId`はUUID |
| ホストURL | `/rooms/{roomId}/host?key={hostKey}` |
| ルーム作成項目 | `title / eventDate / memo / budgetAmount` |
| `budgetAmount` | 任意、0円以上。未入力なら予算差分を非表示 |
| 商品項目 | `name / price / quantity / memo` |
| `price` | 見積単価。UIでは「見積単価」と表示 |
| API状態 | `proposed / accepted / rejected` |
| UI状態 | 提案中／採用／見送り |
| 参加者の権限 | 提案の追加、自分の提案の編集・削除 |
| ホストの権限 | 全提案の編集、採用、見送り、購入チェック |
| 見送りデータ | DBに保持し、通常画面では非表示 |
| メイン合計 | acceptedの `price × quantity` の合計 |
| 補助合計 | proposedの `price × quantity` の合計 |
| 予算差分 | `budgetAmount - acceptedTotalPrice`。負値は超過額として表示 |
| 商品別数量 | acceptedのみ |
| 採用後編集 | 参加者が編集した場合のみproposedへ戻す。ホスト編集はacceptedを維持可 |
| 購入チェック | acceptedのみ変更可能 |
| リアルタイム同期 | MVP外。手動更新で再取得する |
| AI・OCR | 実装しない |
| 実購入額 | acceptedかつ購入済みの商品へ `actualPrice` を記録する |
| 購入者 | 購入済み商品ごとに `paidByParticipantId` を記録する |
| 精算対象者 | ルーム参加者からホストが選択。初期値は全参加者 |
| 割り勘方式 | 実購入額合計を精算対象者で均等割り |
| 端数 | 1円単位。参加登録順の早い人から1円ずつ加算 |
| 立替精算 | 各人の負担額と立替額から最小限の送金案を算出する |
| 精算記録 | 送金案ごとに未精算／精算済みを記録する |
| 精算確定後の変更 | 精算済みが1件でもあれば購入額・購入者・対象者をロック。ホストが全精算をリセットした場合のみ再編集可 |
| 決済 | 実装しない。送金はアプリ外で行う |

### 権限表

| 操作 | 参加者 | ホスト |
| ルーム閲覧 | 可 | 可 |
| 商品提案 | 可 | 可 |
| 自分の提案編集・削除 | 可 | 可 |
| 他人の提案編集 | 不可 | 可 |
| 採用・見送り | 不可 | 可 |
| 購入済みチェック | 不可 | 可 |
| 実購入額・購入者の登録 | 不可 | 可 |
| 精算対象者の選択 | 不可 | 可 |
| 精算済み記録 | 自分が支払側の精算のみ可 | 全件可 |

### トークンとキーの扱い

- 参加API成功時に参加者トークンを発行する
- 参加者トークンは `localStorage` にルーム単位で保存する
- 参加者操作では `X-Participant-Token` ヘッダーを送る
- ホスト画面はURLの `key` を初回読込時に `sessionStorage` へ移し、可能ならアドレスバーからクエリを除去する
- ホスト操作では `X-Host-Key` ヘッダーを送る
- hostKeyをAPIレスポンス、ログ、画面上の通常本文へ再表示しない
- 参加URLとホストURLを明確に分け、ホストURLには共有禁止の警告を出す

## 7. デプロイ先の確定

### フロント候補の比較

| 比較項目 | Cloudflare Pages | Vercel | GitHub Pages |
| Vite SPA | 対応 | 対応 | Actionsでビルドが必要 |
| `/rooms/{uuid}`直リンク | top-level `404.html`がなければSPAとして自動fallback | `vercel.json` rewriteが必要 | 標準fallbackがなく404対策が必要 |
| Git連携 | push・PR連携 | push・PR連携 | GitHub Actionsまたは公開ブランチ |
| モノレポ | frontendをルート指定可能 | frontendをルート指定可能 | workflowで作業ディレクトリ指定 |
| 環境変数 | Pages設定で管理 | Project設定で管理 | Actions Secrets/Variablesでビルド時注入 |
| 設定量 | 少ない | 少ない | 3候補中もっとも多い |
| 今回の評価 | **採用** | 次点 | 不採用 |

### 選定結果

フロントエンドは **Cloudflare Pages** を採用する。

理由:

1. OrderRoomはReact Routerを使う純粋なVite SPAであり、Cloudflare Pagesは該当構成を標準でSPAとして扱える
2. 参加URL `/rooms/{uuid}` やホストURLを直接開く導線で、追加rewriteなしにSPAへ到達できる
3. GitHub Pagesで必要になるActions workflow、リポジトリ配下のbase path、SPA 404対策を避けられる
4. Vercelも十分適合するが、本アプリではFunctionsやSSRを使わないため、Vercel固有機能よりCloudflare Pagesの静的SPA配信の単純さを優先する

将来Next.js、SSR、Vercel Functionsへ移行する場合はVercelを再検討する。GitHub Pagesはデモ用静的サイトには向くが、今回の本番フロントには採用しない。

### 構成

| 領域 | 採用先 | 確定内容 |
| --- | --- | --- |
| フロント | Cloudflare Pages | Vite SPAを配信し、GitHub既定ブランチから自動デプロイ |
| API | Render Web Service | Spring BootをDockerで実行し、GitHub既定ブランチから自動デプロイ |
| DB | Neon PostgreSQL | 本番ブランチを使用し、RenderからSSL接続 |
| ソース | GitHub | `sisicity4/OrderRoom`を唯一のデプロイ元にする |

RenderはJavaをネイティブランタイム対象としていないため、Docker方式に固定する。これによりJava 21と起動コマンドをリポジトリ内で再現可能にする。

### リポジトリへ追加する設定

#### Cloudflare Pages設定

- Root directory: `frontend`
- Build command: `npm run build`
- Build output directory: `dist`
- Production branch: GitHubの既定ブランチ
- top-level `404.html`は作成しない。Cloudflare PagesのSPA fallbackを利用する

#### `backend/orderroom-backend/Dockerfile`

- Java 21でMavenビルドする
- 実行用イメージへjarだけをコピーする
- Spring Bootを `$PORT` で起動する
- RenderのHealth Check Pathを `/api/health` にする

#### Spring Boot本番設定

```properties
server.port=${PORT:8080}
spring.datasource.url=${DB_URL}
spring.datasource.username=${DB_USER}
spring.datasource.password=${DB_PASSWORD}
app.frontend-base-url=${FRONTEND_BASE_URL}
spring.jpa.show-sql=false
```

MVP期間は既存の `ddl-auto=update` を本番でも暫定利用する。v1.0後の最優先技術負債としてFlyway導入と `ddl-auto=validate` への移行をIssue化する。

### 環境変数

| 配置先 | 変数 | 内容 |
| --- | --- | --- |
| Cloudflare Pages | `VITE_API_BASE_URL` | Render APIのHTTPS URL |
| Render | `DB_URL` | NeonのJDBC接続URL。SSL必須 |
| Render | `DB_USER` | NeonのDBユーザー |
| Render | `DB_PASSWORD` | NeonのDBパスワード |
| Render | `FRONTEND_BASE_URL` | Cloudflare Pages本番URLまたは独自ドメイン |
| Render | `SPRING_PROFILES_ACTIVE` | `prod` |

アプリ接続にはNeonのpooled接続を使用し、JDBC URLにSSL設定を含める。DB認証情報をCloudflare Pagesへ置かない。

### CORS

- 本番は `FRONTEND_BASE_URL` と一致するCloudflare Pages本番URLまたは独自ドメインだけを許可する
- PR Previewから本番APIを呼ばせない。必要なら別のstaging APIで `https://*.pages.dev` を許可する
- 開発環境はVite proxyを優先する
- `X-Participant-Token` と `X-Host-Key` を許可ヘッダーに含める
- `*` とcredentialsの併用はしない

### デプロイの受け入れ条件

- Cloudflare Pagesの `/rooms/{uuid}` を直接開いても404にならない
- Renderの `/api/health` が200を返す
- Cloudflare PagesからRender APIを呼べる
- RenderからNeonへSSL接続できる
- 本番ログへDBパスワード、participant token、hostKeyが出ない
- Pages Preview URLを本番CORSへ許可しない

## 13. MVP完成条件

MacとiPhoneなど2台以上の端末で、次を本番環境上で完遂できること。

1. ホストがイベント名、開催日、メモ、任意の予算上限でルームを作る
2. 参加URLをコピーして共有する
3. 別の利用者が共有URLを直接開く
4. 表示名だけ入力して参加する
5. 商品名、見積単価、数量、メモを提案する
6. 自分の提案を編集・削除する
7. ホストが提案を採用または見送りにする
8. 採用品だけを買い物リストへ表示する
9. 採用品に購入済みチェックを付ける
10. 採用済み見積、提案中見積、予算差分、商品別数量を表示する
11. 購入済み商品へ実購入額と購入者を記録する
12. 精算対象者を選び、均等割りの負担額を表示する
13. 複数購入者の立替額から送金案を表示する
14. 支払側本人またはホストが精算済みを記録する
15. 手動更新で別端末の変更を取得する
16. 空状態、読込中、通信失敗、存在しないルームを表示する

## 14. 受け入れ条件

### ルーム作成

- イベント名が必須
- 開催日、メモ、予算上限は任意
- 作成後に参加URLをコピーできる
- ホストURLは作成者だけに表示する
- コピー後に確認表示を出す

### 参加

- 共有URLから直接参加画面を開ける
- ルーム名が表示される
- 表示名だけで参加できる
- 2〜3タップ程度でルームへ到達できる

### 提案

- 商品名、見積単価、数量、メモを入力できる
- 希望者名が表示される
- 自分の提案だけ編集・削除できる
- 参加者が採用品を編集すると提案中へ戻る

### ホスト

- proposedをaccepted / rejectedへ変更できる
- acceptedだけを購入済みにできる
- hostKeyなしではホスト操作ができない

### 集計・予算

- メイン合計はacceptedだけ
- proposedは補助合計として分離する
- 商品別数量はacceptedだけ
- rejectedは含めない
- 予算未入力時は予算差分を表示しない
- 集計値はDBへ保存しない

### 割り勘・立替精算

- 見積額ではなく実購入額だけを精算へ使用する
- 購入者と精算対象者を明示する
- 各参加者の負担額合計と実購入額合計が一致する
- 各参加者の立替差額合計が0円になる
- 端数配分が毎回同じ結果になる
- 実際の送金処理は行わない
- 参加者は自分が支払う精算だけを精算済みにできる
- 精算済みが1件でもある状態では計算元を変更できない
- ホストが精算をリセットすると全件未精算になり、再計算できる
