## 2. 仕様変更

仕様変更を行う前に、次を確認する。

1. 変更理由
2. 対象Sliceと期限への影響
3. FE、BE、画面、テストへの影響
4. MVPへ残すか、将来候補へ送るか
5. 更新が必要な資料

変更後は、一つの資料だけでなくリンク先の専門資料も確認する。

### 変更してはいけない固定事項

- API状態は `proposed / accepted / rejected`
- UI表示は「提案中／採用／見送り」
- 参加者操作はparticipant tokenで確認する
- ホスト操作はhostKeyで確認する
- メイン集計はacceptedの見積
- 精算は実購入額を使用する
- AI、OCR、アプリ内決済はMVP外

変更が必要な場合はPMがロードマップの更新記録へ理由を残す。

## 3. 作業単位

- 1つの作業では一つの利用者操作または一つの受け入れ条件を完成させる
- 関係ない変更を同時に混ぜない
- ファイルの存在ではなく、操作の完遂を作業完了とする
- 作業状態は「未着手」「モックUI」「API実装済み」「結合済み」「受け入れ確認済み」で区別する。
- モックUIやAPI単体実装だけでは、利用者導線の完了とは扱わない。
- MVP外の機能を独断で追加しない。

## 4. GitHub Issueの運用

- MVPに関係する作業は、原則としてGitHub Issueで管理する。
- Issueには少なくとも、担当者、対象Slice、期限、完了条件、依存Issueを記載する。
- 担当者は着手時、進捗に変化があった時、完了時にIssueへコメントする。
- PRには対応Issueを記載し、完了時は `Closes #番号` を使う。
- 期限に影響する未着手・ブロックは、PMがP0/P1/P2を付け、担当再配分またはスコープ縮小を判断する。

## 5. 実装前チェック

- [ ] 対象Sliceが分かる
- [ ] 利用者と権限が分かる
- [ ] 入力と出力が決まっている
- [ ] 空、読込中、失敗、権限不足を確認した
- [ ] 受け入れ条件を説明できる
- [ ] 関係する専門資料が一致している

## 6. レビューで確認すること

### 共通

- 受け入れ条件を再現できる
- 秘密情報を出力していない
- 用語が資料と一致する
- MVP外の機能が混入していない
- 次のSliceを壊していない

### FE

- 375px、390px、430pxで主操作を完了できる
- 読込中、空、通信失敗、404、403がある
- 二重送信を防いでいる
- tokenとhostKeyを混同していない
- 見積額、実購入額、負担額、立替額を区別している

### BE

- 入力値とroom所属を検証している
- participant tokenとhostKeyの権限が分離されている
- エラー形式が共通化されている
- 外部の本番DBなしでテストできる
- 集計と精算の合計が一致する

### QA

- 通常操作と異常操作を確認する
- 別端末、再読込、不正URL、不正tokenを確認する
- 不具合へ再現手順と重大度を付ける

## 7. 完了の定義

作業完了には次を含む。

- 利用者操作が最初から最後まで通る
- 対応する受け入れ条件を確認した
- 自動テストまたは手動確認結果がある
- 未対応事項と既知の制約が明記されている
- 関係する資料が実装と一致している

## 8. 運用上の禁止事項

- 参加者IDだけで本人確認しない
- token、hostKey、DB認証情報をログへ出さない
- rejectedを集計へ含めない
- 見積額を実購入額として精算しない
- 精算済みがある状態で計算元を変更しない
- 旧資料へ独自のAPI仕様を追記しない
- 画面があるだけでSlice完了としない
